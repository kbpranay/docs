# Data and tool integrations

Agentic AI nodes can use six types of tool to reach outside the flow and fetch real data. Each tool has its own configuration — set it up once, then reference it from any agent. This article walks you through each tool type and shows you how to manage flow environment variables.

## Before you begin

- The SQL tool requires a **read-only** database connection string.
- The KB tool requires a DocBrain index. See [DocBrain](/intents-entities/docbrain) for how to create one.
- Make sure you have credentials, URLs, and keys ready before opening the tool configuration dialog.

## Configuring an API tool

Use an **API tool** to call any external service — CRM lookups, ticket creation, payments, translation, and so on.

**Configuration:**

| Field | Description |
|---|---|
| Tool Type | `API` |
| Tool Name | Short identifier. |
| Description | What the tool does. This is what the agent reads to decide when to use it. |
| Method | `GET` or `POST`. |
| URL | The endpoint. |
| Response Type | Expected response format. |
| Timeout | Request timeout. |
| Headers | Name/value pairs. |
| Cookies | Cookie values. |

**Parameters** — each API call can take multiple parameters:

| Field | Description |
|---|---|
| Name | Parameter name. |
| Description | What the agent should put here. |
| Select Type | `string`, `number`, `integer`, or `boolean`. |
| Required | Whether this parameter is mandatory. |
| Query Parameter | If true, appends as `?name=value`. If false, goes in the body. |

## Configuring a Knowledge Base (KB) tool

Use a **KB tool** to let the agent read from a DocBrain or Knowledge Base — similar to a RAG lookup, but available at tool-call granularity.

**Configuration:**

- **Tool Type** — `KB`
- **Tool Name** and **Description**
- **Choose DocBrain** — the knowledge base the tool should query.
- **Index ID** — up to 3 can be shown; a single index ID is selected automatically.
- **Auto-use Latest Index ID** — toggle on to always query the most recent index.

**Advanced settings:**

- **Type** — Semantic or Keyword.
- **Method** — one of three `Gpt_mix` variants.
- **TopK** — the number of results to retrieve.
- **Filtered Search** — filter by Doc ID, Source ID, URL, Original URL, or Doc Type.

## Configuring a Table tool

Use a **Table tool** to query structured data held in a CSV file.

**Configuration:**

1. Upload a **.csv** file (max 5 MB).
2. For each header, set:
   - **Name**
   - **Description**
   - **Match Type** — Exact or Close.
   - **Type** — String, Category, Datetime, Integer, Float, or Boolean.

The agent can then run SQL-like queries against the table using natural language.

## Configuring a File tool

Use a **File tool** to let the agent read content from PDF documents.

**Configuration:**

- PDF format only.
- Up to **5 files**, **5 MB** each, with fewer than **10 pages** per file.

Use the File tool for reference documents your agent needs — policy PDFs, one-pagers, product sheets.

## Configuring a SQL tool

Use a **SQL tool** to run read-only SQL against a connected database.

**Configuration:**

- **DB URL** — this must be a **read-only** connection string.
- For each table the agent can query, define:
  - **Table Name**
  - **Metadata description** — helps the agent understand when and how to query the table.

<Callout type="warning">
  **Warning:** The SQL tool only accepts **read-only** DB URLs. Don't give it a connection that can mutate your data — the agent's query plans are probabilistic and could corrupt state if the database is writeable.
</Callout>

## Configuring a Webpage tool

Use a **Webpage tool** to fetch and read live webpage content. This is ideal for pages that change frequently — pricing pages, offer pages, status dashboards.

**Configuration:**

- **Tool Name** and **Description**
- **Webpage URL**
- **Cache Minutes** — how long the cached content should be reused before a fresh fetch, so you don't hammer the target site.

Use this tool for pages where the content changes in minutes or hours, but re-fetching on every query would be wasteful.

## Using environment variables

Flow environment variables are reusable values — API keys, config values, default parameters — accessible across every node in the current flow.

### Defining a variable

1. From the workspace header, click the **⚙ Flow Environment Variables** icon.
2. In the **Flow Env Variable** dialog, click **+ Add**.
3. Enter a **Name**, **Value**, and **Type** for the variable.
4. Click **Save**.

### Referencing a variable from a node

| Node type | How to reference the variable |
|---|---|
| **Change** | Manipulate the value through the Change node's operations using the `env.*` path. |
| **Switch** | Use the value in a conditional branch. |
| **Template** | Use `{{env.Name}}` for dynamic substitution. |
| **Function** | Use `env.get("Name")` inside your custom JavaScript. |

<Callout>
  **Tip:** Keep API keys and secrets in environment variables rather than hardcoding them into nodes. Environment variables are the right layer for values that change per environment — dev, staging, production.
</Callout>
