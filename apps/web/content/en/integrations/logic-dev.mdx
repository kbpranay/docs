# Start, logic, and dev nodes

**Start** nodes are how a flow begins. **Logic** nodes branch and transform data as it moves through the flow. **Dev** nodes connect the flow to external systems and help you debug while you're building. **Link** nodes stitch multiple flows into a single composition.

## Before you begin

- You should have a flow open in the workspace. See [Managing flows in AIDA](/integrations/flows-basics).
- If you're planning to trigger this flow from an external system, know the URL, method, and authentication the caller will use.

## Starting a flow

Every flow begins with a Start node. You can start a flow manually or expose it over HTTP as an API endpoint.

### Start node

Use the **Start** node to trigger a flow manually by injecting data. This is typically used for testing, or when the flow is kicked off from inside AIDA rather than from an external caller.

**Settings:**

- **Input type** — click the timestamp dropdown to choose `string`, `number`, or another supported type.
- **Repeat function** — re-emit the same message on a fixed interval. Use this for health checks or scheduled triggers.

### Trigger via API

Use **Trigger via API** to let external systems kick off a flow by calling an endpoint. This is how you integrate AIDA flows into webhooks, cross-system automations, and existing orchestration platforms.

**Settings:**

| Field | Options |
|---|---|
| **Type** | REST or SOAP |
| **Method** | GET, POST, PUT, or PATCH |
| **URL** | The endpoint your external caller will hit |

<Callout type="info">
  **Tip:** Pair a **Trigger via API** node at the start of your flow with an **API response** node (see below) at the end. The API response node sends the flow's output back to the caller synchronously.
</Callout>

## Branching and transforming data

Logic nodes let you make decisions and reshape the data passing through the flow.

### If/Else

Use the **If/Else** node to evaluate rules against incoming data and forward to the matching output.

You can match on:

- A message property
- Flow context or global context
- An environment variable
- A JSONata expression

Rules evaluate top to bottom. You can optionally **stop at the first match**; otherwise, every matching branch fires.

### Change

Use the **Change** node to transform data in place before passing the message forward.

**Operations:**

| Operation | Effect |
|---|---|
| **Set** | Assign or update a value. |
| **Change** | Transform an existing value. |
| **Move** | Relocate a value from one property to another. |
| **Delete** | Remove a property. |

### Template

Use the **Template** node to generate a dynamic string from `msg.payload`. By default it uses **Mustache** templating, but you can switch Mustache off if you want raw output.

The node takes the incoming `msg` object plus a template string and outputs a `msg` with the populated property.

**Example:** `Hello {{name}}, your order {{orderId}} is confirmed.`

## Using dev nodes

Dev nodes are listed in the order they appear in the palette.

### Comment

Use the **Comment** node to leave non-executing annotations on the canvas. Drag it in from the Dev palette, then double-click to edit the text.

<Callout>
  **Tip:** Use comments liberally. Future-you — and your teammates — will thank present-you when someone has to debug this flow six months from now.
</Callout>

### REST API

Use the **REST API** node to call any HTTP endpoint from a flow.

**Supported methods:** GET, POST, PUT, DELETE.

**Authentication options:**

| Auth | Fields |
|---|---|
| **Basic** | Username + Password |
| **Bearer** | Token |
| **OAuth2** | Grant type (Client credentials, Password, and others), Access token URL, Client ID, Client Secret, Scope |

**Query parameters** are key/value pairs appended to the URL for filtering, sorting, or pagination.

### Function

Use the **Function** node to run custom JavaScript. The incoming message is passed in as a `msg` object, with the body on `msg.payload`. Use this for transformations too complex for Change or Template, or when you need to call Node.js utilities.

### Delay

Use the **Delay** node to pause the flow or rate-limit message throughput.

**Properties:**

- **Delay (ms)** — how long to wait. This can be overridden by a property on the incoming `msg`.
- **Reset** — clear all held messages.
- **Flush** — send all held messages immediately.

### API response

Use the **API response** node to handle the response sent back from an API call. This node is used mostly in flows that were triggered via API — see the [Trigger via API](#trigger-via-api) option on the Start node.

### Debug

Use the **Debug** node to log `msg.payload` (the primary data) and the full `msg` object (metadata) so you can inspect state while you're developing.

<Callout>
  **Tip:** Leave Debug nodes in complex flows while you're iterating. You can disable their output with a single click without deleting them — and re-enable when you need to inspect again.
</Callout>

## Linking flows together

Use the **Link** palette to stitch flows into a single composition.

- **Flow In** — an entry point that lets data, signals, or control pass between nodes in different flows. Pair it with a Flow Out in another flow to build modular, reusable automations.
- **Flow Out** — the exit point paired with a Flow In.

Use Flow In / Flow Out pairs when you want to:

- Reuse common logic across multiple flows.
- Break a complex flow into digestible pieces.
- Test individual sections of a flow in isolation.
